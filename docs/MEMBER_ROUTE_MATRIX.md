# Member Route, API, Feature, and State Matrix

Updated: 2026-07-16

Scope: every `page.tsx` route under `apps/web-member/app`

Owners are repository delivery owners, not production on-call assignments. `ready` means the frontend currently calls an implemented contract; it does not certify every edge state.

State vocabulary is fixed to: `loading`, `empty`, `partial-failure`, `error`, `offline`, `stale`, `maintenance`, `success`, `retry`, and `session-expired`. Every state must appear exactly once under Covered, Known gaps, or N/A. The repository audit enforces route completeness, ownership metadata, and state classification.

## Ownership and closure policy

- P0/P1 changes must name both the frontend delivery owner and API owner/readiness in this matrix.
- Money, identity, and session changes require automated checks plus a rollback note in the change or evidence log.
- Known gaps are acceptance work, not claims that the API is absent.
- Add or remove a route only together with this matrix; `pnpm audit:member-route-matrix` fails on drift.
